`#options-host` at desktop widths while `.transform-session-actions` is sticky-pinned over it, and the
compact wrap rule starts only at ≤1023 px. One layout fix (wrap/shrink strategy at desktop widths +
non-overlapping placement for pinned essentials) likely resolves all four.

**F-001 and F-002 are the probable source of the owner's "layouts overlapping / buttons not working"

Free Transform clicking it cancels the user's session.

## Method notes (geometry sweep, Task 3)

- The audit browser pane runs a hidden renderer: no paint, no screenshots, and `requestAnimationFrame`
  never fires — which suspends the app's DirtyFlag→rAF render batching entirely. Restored via a
  query-gated fallback in `index.html` (`?audit-raf` swaps rAF for a 16 ms timeout with identical
  async-batch semantics). Production behavior untouched without the flag. **Triage decision needed:
  keep as a testability affordance or revert after the audit.**
- `.claude/launch.json` declared port 5173 but Vite runs on port 3000; the preview opened a dead
  URL. Fixed in this audit (port 3000).
- Probe false-positive classes identified and excluded from findings: (a) content inside
  `overflow: auto` containers (properties dock, scrollH 665 vs clientH 346) reports as "occluded"
  when merely scrolled below the fold — cleared functionally: after scrolling the dock (scrollTop
  319) the rotation field hit-tests to itself; (b) at 375 px the layout stacks and the page
  scrolls, so below-the-fold controls are not "clipped".
- Zero surface-container violations (`.appbar`/`.rail`/docks/`.statusbar`) in every state at
  1280×800, 1024×800, and 375×812. All overlap findings are content-level occlusion/clipping.

## Method notes (behavior sweep, Tasks 4–7)

- The hidden renderer defers `canvas.toBlob` callbacks by up to several seconds; save/export
  evidence collection needs ≥2.5 s waits. Early "missing download" observations were timing
  artifacts, not app bugs — final evidence is conclusive.
- Reading `style.transform` goes through CSSOM serialization, which rounds `scale(1.0000000000000002)`
  to `scale(1)` — zoom drift was verified via behavior (pan-reset firing or not), not via strings.
- The audit browser profile is ephemeral per preview-server restart, so autosave persistence across
  real browser restarts could not be observed; the within-session cycle (debounced IDB write →
  reload → restore offer → restore) is fully verified. The startup restore-offer toast
  auto-dismisses after 10 s, shorter than harness round-trip latency — verified by invoking the
  exported `tryRestoreOffer()` directly (identical code path to the `src/main.ts` startup call).
- Canvas gestures used synthetic PointerEvents with a try/catch wrapper around
  `setPointerCapture` (synthetic pointers are never "active"); capture only redirects real pointer
  streams, so drag logic is unaffected. The app's single-pointer routing correctly ignores new
  pointerdowns while another pointer is held.

# Broken

## F-005 — [broken] Custom background color picker can never become visible

- Surface: Options bar, workspace settings (`#bg-color-picker`)
- State: any viewport, Background = Custom
- Repro: 1) Click the "Custom" background theme button 2) Look for the color picker
- Expected: `syncBackgroundUI()` shows the picker (`style.display = 'inline-block'`) so the user
  can choose the custom color
- Actual: the input keeps the `hidden` HTML attribute (index.html) and `src/style.css` declares
  `[hidden] { display: none !important; }` — the `!important` author rule defeats the inline
  style, so the picker stays `display: none` (0×0) forever. Custom mode is stuck on the stored
  `bgColor` (default `#ffffff`, indistinguishable from White). The input handler itself works —
  a programmatic `input` event recolors the viewport — so only visibility is broken.
- Candidate cause: `index.html` `hidden` attribute + `[hidden]` rule in `src/style.css`;
  `syncBackgroundUI()` in `src/canvas.ts` should toggle the attribute, not inline display

## F-008 — [broken] "Show controls" toggle has no immediate effect

- Surface: Options bar (Move tool, `show-controls` toggle)
- State: Move tool, any layer selected
- Repro: 1) Select a layer (handles visible) 2) Toggle "Show controls" off 3) Observe the canvas
- Expected: transform handles disappear immediately (and reappear on toggling back)
- Actual: nothing changes until an unrelated action triggers a composite; the flag itself works
  (pixel evidence: corner overlay sum 25064 with handles, 5697 without — but only after a forced
  `notify('composite')`). The snap toggle calls `notify('composite')` in its setter; this one
  does not.
- Candidate cause: `setShowTransformControls()` in `src/canvas-overlay.ts` missing a composite
  notification (`src/tools/move.ts` wires it directly as the option setter)

## F-010 — [broken] All keyboard shortcuts go dead while a button or link keeps focus

- Surface: Keyboard routing (`src/main.ts` + `isInteractiveTarget`)
- State: any — occurs after clicking/tabbing to any button (rail, appbar, toggles…) since focus
  stays on it
- Repro: 1) Click (or Tab to) any toolbar button so it keeps focus 2) Press V/H/Z/C, Ctrl+T,
  Ctrl+Z, or Space
- Expected: single-letter tool shortcuts and history/session shortcuts work unless the user is
  typing in a text field
- Actual: `isInteractiveTarget()` includes `BUTTON` and `A`, so every document-level shortcut is
  ignored until focus moves to a non-interactive element (verified: `h` does nothing while
  `#btn-save` is focused, works immediately after blur). Buttons don't consume letter keys, so
  the protection is only needed for typing contexts (INPUT/TEXTAREA/SELECT/contentEditable).
- Candidate cause: over-broad tag list in `isInteractiveTarget()`
  (`src/transform-session-guard.ts:21`)

## F-012 — [broken] Escape in a properties transform field leaves the stale draft visible

- Surface: Properties dock transform fields (`#prop-transform-x/y/width/height/rotation`)
- State: any layer selected
- Repro: 1) Focus a transform field (e.g. Y showing 480) 2) Type a different value (999)
  3) Press Escape
- Expected: the field reverts to the document value (480) and blurs
- Actual: the document value is correctly untouched and the field blurs, but the field keeps
  displaying the abandoned draft (999) until an unrelated panel sync rewrites it. Cause: the
  Escape handler calls `syncTransformFields()` while the input is still focused, and that sync
  skips `document.activeElement` to protect live typing — then blurs with no re-sync. The
  options-bar inputs assign `input.value = String(opt.get())` directly and do not have this bug.
- Candidate cause: Escape branch in `initPropertiesPanel()` (`src/properties-panel.ts`) —
  sync-then-blur ordering

## F-013 — [broken] History-panel rows bypass the live-session block

- Surface: History panel (`#history-list` rows)
- State: explicit Free Transform (or crop) session live
- Repro: 1) Select a layer, Ctrl+T 2) Open the History tab 3) Click an older history entry
- Expected: history navigation is blocked while any editing session is live — the undo/redo
  buttons are disabled and Ctrl+Z is inert in this state (both verified)
- Actual: the row click executes `history.jump(i)` and moves the history cursor while the session
  stays live (verified: cursor moved, session still active) — the exact desync hazard the
  session-block exists to prevent ("a mid-gesture undo would desync cached snap candidates and can
  silently abandon the user's in-progress edit"). The `tests/ui-layout.test.mjs` contract asserts
  the block for `src/main.ts` keyboard/buttons but does not cover the history-panel rows.
- Candidate cause: row click handler in `initHistoryPanel()` (`src/history-panel.ts`) calls
  `history.jump(i)` without a `historySessionBlocked()` guard

# Overlap

## F-001 — [overlap] Sticky session actions cover the Move rotation input and snap toggle

- Surface: Options bar (`#options-host`), Move tool
- State: 1280×800, any panel state; worst during explicit Free Transform
- Repro: 1) Open the app at 1280×800 2) Move tool active, layer selected 3) Observe
  `.transform-session-actions` (Apply/Cancel, `position: sticky; right: 70px; z-index: 2`, opaque
  background) pinned over the options row 4) Ctrl+T to start Free Transform 5) Click the rotation
  input
- Expected: every visible option is clickable; session actions never occlude sibling controls
- Actual: Move's option row (scrollWidth 1077) overflows the host (clientWidth 865); the sticky
  Apply (x 742–813) and Cancel (x 806–883) sit on top of the rotation input (x 792–848 — fully
  covered) and the snap area. Disabled action buttons keep `pointer-events: auto`, so the rotation
  input is dead at its center even with no session; with a live session, the click lands on the
  enabled **Cancel** and destroys the user's transform (verified: `elementFromPoint` at the rotation
  input center → `#tool-option-cancel` svg, `applyEnabled: true`).
- Candidate cause: `.transform-session-actions` sticky overlay in `src/style.css` + Move option row
  wider than the host at desktop widths

## F-002 — [overlap] Move geometry options are scroll-only at desktop widths; misclicks land on background toggles

- Surface: Options bar (`#options-host`), Move tool
- State: 1280×800 (snap partially out of view) and 1024×800 (height/link/rotation/snap fully out)
- Repro: 1) Open at 1024×800 2) Move tool 3) Try to click the rotation or snap option
- Expected: at 1024 px the spec requires zero overlap violations — geometry options visible or
  wrapped (the compact wrap rule exists but only applies at ≤1023 px; at exactly 1024 the row is
  `nowrap`)
- Actual: host clientWidth 609 vs scrollWidth 1077 — `height` (partially), `link`, `rotation`, and
  `snap` render outside the visible host with no wrap; the screen space where they "should" be is
  occupied by the workspace-settings **background theme buttons**, so clicking there recolors the
  canvas background instead (verified: rotation center → `button.btn-theme`; link center →
  `span.control-label`). The only access is an undiscoverable horizontal scroll inside a 44 px bar.
- Candidate cause: `@media (max-width: 1023px)` wrap breakpoint too low + no `flex-wrap` or
  min-width handling in the desktop options row (`src/style.css`)

## F-003 — [overlap] Crop options overflow at 1024; sticky crop actions overlap crop-reset and each other

- Surface: Options bar (`#options-host`), Crop tool session
- State: 1024×800, crop session active
- Repro: 1) Open at 1024×800 2) Press C (crop session auto-begins) 3) Inspect the options row
- Expected: crop controls all reachable; Apply/Cancel never overlap Reset or one another
- Actual: crop row scrollWidth 759 vs clientWidth 609; `#tool-option-crop-reset` center hit-tests to

- Candidate cause: same sticky-overlay mechanism as F-001 applied to both crop essential actions

# Cosmetic

## F-006 — [cosmetic] Pan-reset at 100% zoom is floating-point nondeterministic

- Surface: Canvas zoom pill (`#zoom-in` / `#zoom-out`)
- State: any viewport, after panning
- Repro: 1) Pan the canvas (Hand tool or Space-drag) 2) Zoom in 3× via `#zoom-in` 3) Zoom out 3×
  via `#zoom-out` — readout shows 100% but the view stays panned. Repeat with 5×/5× — the view
  snaps back to center.
- Expected: returning to 100% behaves consistently regardless of how many steps got there
- Actual: `setZoom()` resets pan only on `zoom === 1` exactly; the button path accumulates FP
  error (`zoom * (1 ± 0.1/zoom)`), so some cycles land on exactly 1.0 (pan resets) and others on
  1 ± ε (readout rounds to 100%, pan persists). Verified: 5-in/5-out → `translate(0px, 0px)`;
  3-in/3-out → `translate(50px, 30px)` retained; exact `zoomAt(2)`+`zoomAt(0.5)` → resets.
- Candidate cause: exact-equality check in `setZoom()` (`src/canvas.ts:29`) — needs an epsilon or
  step-count-based zoom model

## F-007 — [cosmetic] Zoom tool's options-bar readout goes stale while zooming

- Surface: Options bar (Zoom tool display option)
- State: Zoom tool active
- Repro: 1) Press Z 2) Click the canvas to zoom in 3) Compare the options-bar percent with the
  zoom pill readout
- Expected: both readouts track the live zoom
- Actual: the zoom pill updates (125%) but the options-bar display keeps its render-time value
  (100%) — zoom changes are view-only (no DirtyFlag), and the options bar only re-renders on tool
  change or document dirty flags.
- Candidate cause: `src/tools/zoom.ts` display option + no re-render trigger on zoom in
  `src/options-bar.ts` / `src/canvas.ts`

## F-009 — [cosmetic] Applying crop via the options button strands a session-less Crop tool

- Surface: Options bar (Crop tool session actions) + keyboard routing
- State: Crop session applied via the `crop-apply` button (not Enter)
- Repro: 1) Press C (session opens) 2) Click Apply in the options bar 3) Press C again
- Expected: consistent post-apply state — Enter-apply returns to Move; the button should match,
  or at least C should reopen a session
- Actual: after button-apply the Crop tool stays active with no session; pressing C is a no-op
  (tool already active, so no toolchange fires `beginCrop()`); only clicking the canvas reopens a
  session. Enter-apply cleanly returns to Move.
- Candidate cause: `crop-apply` option in `src/tools/crop.ts` runs `applyCrop()` without the
  `setActiveTool('move')` the keyboard path performs in `src/main.ts`
